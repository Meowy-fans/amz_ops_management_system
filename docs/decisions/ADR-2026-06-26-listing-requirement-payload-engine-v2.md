# ADR-2026-06-26: Build Listing Requirement & Payload Engine V2

**Date**: 2026-06-26
**Status**: Proposed

## Context

The current API-native listing flow relies on a flat attribute pipeline:

- `AmazonSchemaService` exposes required attributes as `List[str]` top-level names.
- Conditional `if/then/else.required` branches are not evaluated against the
  current listing payload.
- Object attributes are considered covered when the top-level payload is
  non-empty; required children are not validated locally.
- Confidence scoring and pending review operate at top-level attribute keys.

This is enough for the current live categories that have been hardened with
rules, learned feedback, and strict-preview tests, but it does not scale to
Amazon product types with deep object schemas such as CHAIR. Static expansion of
all conditional required fields is not acceptable because it would turn many
context-specific offer, hazmat, battery, package, and variation fields into
global blockers.

The replacement must preserve the current safety mechanisms:

- evidence-grounded confidence scoring
- safe-default whitelist
- AI/human review routing
- pending review persistence and deterministic resume
- strict dry-run `VALIDATION_PREVIEW`
- fail-closed behavior for uncertain required facts
- dry-run default and explicit LIVE writes

## Decision

We will build `Listing Requirement & Payload Engine V2` as a new engine that can
replace the current flat attribute resolver, payload renderer, coverage gate, and
review-routing boundary.

V2 will use three explicit intermediate products:

1. `RequirementTree`: the applicable Amazon schema requirement tree for the
   current listing scenario.
2. `ResolutionTree`: path-level values, sources, evidence, confidence, review
   state, and blocking state.
3. `PayloadBuildPlan`: final Listings Items API attributes plus requirement,
   resolution, coverage, review, and audit metadata.

V2 will start read-only, then run in shadow mode, then strict-preview mode, and
only then replace the existing production path.

The activation contract is:

```text
LISTING_PAYLOAD_ENGINE=v1|v2|shadow
```

V2 must not remove or rewrite the existing live path until CABINET, HOME_MIRROR,
and OTTOMAN strict-preview regressions prove parity.

## Key Contracts

### Conditional Evaluation

V2 will evaluate conditional required fields against the current listing context
and candidate payload. It will not statically merge every `then.required` field.

Because schema conditions can depend on values generated by the engine itself,
V2 will use iterative evaluation:

1. Build initial requirements from always-required fields and conditions that
   can be evaluated from existing draft/context fields.
2. Resolve known paths.
3. Render a candidate payload.
4. Re-evaluate conditions against the candidate payload.
5. Add newly applicable requirements and repeat until stable or an explicit
   iteration limit is reached.

The first implementation must define a small explicit iteration limit. Failure
to converge creates an audit finding and blocks only the affected uncertain
branch. Unsupported condition shapes are visible in condition trace output and
must not become product-type-wide required fields.

### Stable Path Keys

V2 review and override keys must be stable across resume:

- object children use dot paths, such as `frame.color`.
- array object entries use schema selectors when available.
- selector-less arrays must use a deterministic fingerprint or be marked
  non-resumable and fail closed.
- path-key schema/version must be recorded in `PayloadBuildPlan` audit metadata.

### Review and Confidence

V2 will score and review values at path level, then aggregate parent state
conservatively. A parent object is complete only when its required children are
complete and allowed by review/default policy.

Review overrides must replay by stable path key without rerunning LLM extraction.

### Safe Defaults and Sensitive Fields

Safe defaults remain opt-in and path-specific. Defaults that are safe at a
top-level attribute do not automatically apply to child paths.

Sensitive paths for identifiers, compliance, certification, hazmat, and related
regulatory fields remain blocked from LLM/default auto-fill unless explicitly
approved.

### Cutover

V2 will replace only the attribute requirement/resolution/rendering/coverage
boundary. It will continue to integrate with the existing commercial gate,
variation resolver, image selector, quality gate, and Amazon submitter.

## Alternatives Considered

| Option | Pros | Cons | Why Not Chosen |
| --- | --- | --- | --- |
| Patch current flat resolver | Smaller first diff; less new code | Forces tree semantics into flat objects; high regression and review-resume risk | Object payloads and child review need a tree model. |
| Statically expand all conditional required fields | Simple implementation | Severe overblocking for context-specific offer, hazmat, battery, package, and variation branches | Does not reflect Amazon schema semantics. |
| Build a simplified sidecar payload generator | Faster prototype | Drops confidence/review/safe-default/strict-preview parity | User requirement is full replacement, not a simplified generator. |
| Build V2 replacement engine | Clean contracts; supports tree requirements and path-level review | Larger Epic; requires shadow and cutover work | Chosen because it matches long-term replacement needs. |

## Consequences

**Positive**:

- Conditional required fields become explainable per listing scenario.
- Object and nested object payloads can be generated and validated generically.
- Confidence scoring and review can move from top-level attributes to stable
  schema paths.
- Existing live path can remain untouched until V2 parity is proven.

**Neutral**:

- For a period, V2 will exist beside the current pipeline in read-only/shadow
  mode.
- Additional module design and contract tests are required before coding the
  evaluator.
- Some Amazon schema constructs will be intentionally unsupported at first and
  fail closed with audit traces.

**Negative**:

- Larger implementation scope than a targeted CHAIR fix.
- Path-key and condition-convergence mistakes could break pending review resume.
- Full cutover requires strict-preview regression across live categories.
